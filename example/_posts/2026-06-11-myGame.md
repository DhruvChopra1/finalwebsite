I recently finished building a complete multi-game arcade experience in Processing Java. The project is called **Arcade Escape**, and it's a collection of four mini-games all tied together inside a crumbling arcade hub world. The player can walk around, enter different game rooms, collect keys, and ultimately unlock a final credits sequence.

This post is a detailed walkthrough of how the major systems work — from the file I/O that reads the player's name, to the physics engine behind the platformer, to the stealth detection logic.

---

# Full Project Breakdown

## Project Overview

The game is structured around a central **screen state system**. Every major mode — main menu, hub world, each mini-game, game over, and end credits — is controlled by a single integer variable:

```java
int screen = 0;
```

The `draw()` loop checks this variable every frame and routes to the correct function:

```java
void draw() {
  if (screen == 0) { image(mainMenu, 0, 0); }
  else if (screen == 1) { playHubWorld(); }
  else if (screen == 2) { playFlood(); }
  else if (screen == 3) { playWreckageRise(); }
  else if (screen == 4) { playFatalLane(); }
  else if (screen == 5) { playStealth(); }
  else if (screen == 6) { endGame(); }
  else if (screen == 7) { playEndScreen(); }
}
```

This is clean and scalable — adding a new game just means writing a new `playX()` function and adding one `else if`.

## Part 1: File I/O — Player Name System

One of the first things the game does is ask for the player's name using a `JOptionPane` dialog, write it to a text file, and read it back out. This name gets woven into the final credits sequence.

```java
// 1. Ask for the name
name = JOptionPane.showInputDialog("Please enter your name:");

// 2. Handle cancel or blank input
if (name != null && name.length() > 0) {
  println("Hello, " + name + "!");
} else {
  name = "Player 1";
}

// 3. Write it to a text file
output = createWriter("positions.txt");
output.println(name);
output.close();

// 4. Read it back out
reader = createReader("positions.txt");
try {
  line = reader.readLine();
} catch (IOException e) {
  e.printStackTrace();
  line = null;
}
```

**Why write and immediately read back?** This demonstrates the full file I/O cycle — not just storing the name in memory, but persisting it to disk. The `line` variable (read from file) is used later in the end screen, keeping the two concerns separate.

**The null guard** (`if (name != null && name.length() > 0)`) is essential. If the user hits "Cancel" on the dialog, `JOptionPane` returns `null`, which would crash the file writer.

## Part 2: The Hub World — Movement, Animation, and Wall Collisions

The hub world is the connective tissue of the whole game. The player walks around a crumbling arcade, and the environment visually degrades over time using a state machine tied to a global timer.

### The Crumbling Arcade State Machine

```java
globalTimer++;
if (globalTimer > 2700) {
  arcadeFloorState++;
  globalTimer = 0;
}

if (arcadeFloorState == 0)      { image(hubNormal, 0, 0); }
else if (arcadeFloorState == 1) { image(hubCracked1, 0, 0); }
else if (arcadeFloorState == 2) { image(hubCracked2, 0, 0); }
else if (arcadeFloorState == 3) { image(hubCracked3, 0, 0); }
else if (arcadeFloorState == 4) { image(hubCracked4, 0, 0); }
else if (arcadeFloorState == 5) { screen = 6; } // Game Over!
```

Every 2700 frames (roughly 45 seconds at 60fps), the arcade advances to its next cracked state. If the player takes too long, they lose — creating a constant time pressure even in the hub.

### Flipbook Walking Animation

The player character uses a two-frame flipbook animation that cycles while moving:

```java
if (isWalking == true) {
  animTimer++;
  if (animTimer > 20) {
    currentFrame++;
    if (currentFrame > 1) { currentFrame = 0; }
    animTimer = 0;
  }

  if (isMovingLeft)       { currentHubSprite = walkWest[currentFrame]; }
  else if (isMovingRight) { currentHubSprite = walkEast[currentFrame]; }
  else if (isMovingUp)    { currentHubSprite = walkNorth[currentFrame]; }
  else if (isMovingDown)  { currentHubSprite = walkSouth[currentFrame]; }
}
```

Each direction has a two-image array (`walkNorth[0]`, `walkNorth[1]`, etc.) loaded in `setup()`. `animTimer` counts frames and flips the index every 20 frames, creating a walking illusion.

### Axis-Separated Collision (The "Slide" Fix)

A common beginner mistake is checking collision for `(nextX, nextY)` together, which causes the player to get completely stuck on corners. This game solves it by testing each axis independently:

```java
if (checkWall(nextX, hubPlayerY) == false) { hubPlayerX = nextX; }
if (checkWall(hubPlayerX, nextY) == false) { hubPlayerY = nextY; }
```

If only the X movement causes a collision, Y movement still goes through — the player slides along the wall instead of stopping dead. This is a standard technique in 2D game development.

## Part 3: Escape the Flood — Platformer Physics

The flood game is a rising-water platformer. The player must climb a series of platforms before the water catches them.

### Gravity and Landing

```java
// Gravity: accelerate downward every frame
playerVY += 0.6;
playerY += playerVY;
isGrounded = false;

// Platform collision loop
for (int i = 0; i < totalPlatforms; i++) {
  if (playerVY > 0) { // Only check landing when falling
    if (playerX + (floodPlayerW * 0.8) > platformX[i] &&
        playerX + (floodPlayerW * 0.2) < platformX[i] + platformW[i] &&
        playerY + floodPlayerH >= platformY[i] &&
        playerY + floodPlayerH <= platformY[i] + platformH[i] + 15) {

        playerVY = 0;
        isGrounded = true;
        playerY = platformY[i] - floodPlayerH;
    }
  }
}
```

A few design choices worth noting here. The `playerVY > 0` guard means the game only checks for landing when the player is moving downward — this prevents being "caught" on a platform from below while jumping. The horizontal check uses 20%–80% of the player's width rather than the full edges, giving a little forgiveness on landings. And when landing, `playerY` is snapped precisely to the top of the platform to prevent sinking.

### The Rising Water Hazard

```java
waterY -= 0.3; // Water rises every frame

fill(0, 100, 255, 150);
rect(0, waterY, width, height - waterY);

if (playerY + (floodPlayerH / 2) > waterY) {
  floodLives--;
  if (floodLives <= 0) {
    screen = 6; // Game Over
  } else {
    playerX = 50;
    playerY = height - 100;
    playerVY = 0;
    waterY = height + 50; // Reset water on respawn
  }
}
```

The water starts off-screen below and slowly creeps upward. The collision check uses the player's vertical midpoint rather than their feet — this gives a small grace period before death, making the game feel fair rather than cheap. On losing a life (but not the last one), the water resets to give the player a fresh chance.

## Part 4: Wreckage Rise — Vertical Scroller with Score Scaling

Wreckage Rise is a vertical survival game where the player jumps between falling debris platforms while dodging enemies and collecting chocolate bars.

### Score-Based Difficulty Scaling

```java
if (frameTimer > 60) {
  updateScore(0); // +1 point per second
  frameTimer = 0;

  if (score >= 1  && activeEnemies == 1) { activeEnemies = 2; /* spawn enemy */ }
  if (score >= 30 && activeEnemies == 2) { activeEnemies = 3; /* spawn enemy */ }
  if (score >= 60 && activeEnemies == 3) { activeEnemies = 4; /* spawn enemy */ }

  if (score >= 20 && activeChocolate == 0) { activeChocolate = 1; /* spawn pickup */ }
  if (score >= 50 && activeChocolate == 1) { activeChocolate = 2; /* spawn pickup */ }
}
```

Rather than hardcoding a difficulty level, the game uses survival time (converted to score) as the trigger for scaling. New enemies and pickups are introduced at specific thresholds, giving the player time to adjust before each escalation.

### The Jump Physics

```java
image(playerSprite, playerX, playerY);
playerY += playerSpeed;

// Gravity curve
if (playerSpeed < 7) {
  playerSpeed += 0.7;
}

// Platform bounce triggers a jump
boolean checkCollisiondebris() {
  for (int i = 0; i < debrisX.length; i++) {
    if (playerX + 85 >= debrisX[i] && playerX + 15 <= debrisX[i] + 100 &&
        playerY + 70 >= debrisY[i] && playerY + 105 <= debrisY[i] + 60) {
      return true;
    }
  }
  return false;
}

// In the main loop:
if (checkCollisiondebris() && playerSpeed > 0) {
  playerSpeed = -25; // Launch upward on platform hit
}
```

Unlike a traditional jump triggered by pressing a button, here the *platforms* trigger the jump automatically when the player lands on them. The player's job is to position themselves correctly — the game handles the upward velocity. Gravity is applied as an acceleration curve (capped at 7) rather than a fixed fall speed, giving the arc a natural feel.

## Part 5: Fatal Lane — Mouse-Controlled Dodge Game

Fatal Lane is a traffic dodging game where the player controls a car with the mouse. Holding left-click accelerates, increasing both the player's speed and the traffic's speed.

### Collision Using Distance

Instead of a rectangle-overlap check, this game uses distance between center points for collision:

```java
void collision() {
  for (int i = 0; i < 4; i++) {
    int herocenter = heroCarX + 40;   // Center of 80px hero car
    int heroY = 835;

    float carCenterX = carsX[i];
    float carCenterY = carsY[i] + 75; // Center of 150px traffic car

    int collisionDist = int(dist(herocenter, heroY, carCenterX, carCenterY));

    if (collisionDist < 65) {
      carHit = true;
    }
  }
}
```

Using `dist()` creates a circular hitbox, which tends to feel more forgiving and accurate for vehicles than a tight rectangle. The radius of 65 pixels was tuned to visually match the car sprites.

### Speed Boost Mechanic

```java
void randomDrop() {
  for (int i = 0; i < 4; i++) {
    float currentSpeed = carSpeed[i];

    if (mousePressed && mouseButton == LEFT) {
      currentSpeed += 8; // Traffic speeds up with the player
    }

    carsY[i] += currentSpeed;
    // ...
  }
}

// Distance also increases faster while holding click:
if (mousePressed && mouseButton == LEFT) {
  currentDist += 2;
} else {
  currentDist += 1;
}
```

Holding left-click doubles progress toward the 5000m goal, but also makes traffic faster — a risk/reward trade-off that gives skilled players a way to finish the game quicker.

## Part 6: Precision Heist — Stealth with Lives

The stealth game challenges the player to retrieve a document from a randomized location and return to the exit without being caught by patrolling guards.

### Guard Patrol System

Guards are controlled by a parallel arrays system. Each index `i` represents one guard with its own position, speed, patrol bounds, and movement axis:

```java
int[] guardX     = {130, 1100, 1070, 1040, 870};
int[] guardY     = {170, 550,  200,  830,  190};
int[] guardSpeed = {3,   3,    3,    3,    3};
boolean[] VertorHor = {true, false, true, false, false};
int[] guardMinX  = {0, 40, 0, 110, 300};
int[] guardMaxX  = {1200, 1120, 1200, 1120, 870};

void guardControl() {
  for (int i = 0; i < 5; i++) {
    if (millis() > guardTimings[i]) { // Staggered entry timing
      if (VertorHor[i] == true) {
        guardY[i] += guardSpeed[i];
        if (guardY[i] < guardMinY[i] || guardY[i] > guardMaxY[i]) {
          guardSpeed[i] *= -1; // Reverse direction at bounds
        }
      } else {
        guardX[i] += guardSpeed[i];
        if (guardX[i] < guardMinX[i] || guardX[i] > guardMaxX[i]) {
          guardSpeed[i] *= -1;
        }
      }
      image(guard, guardX[i], guardY[i]);
    }
  }
}
```

The `guardTimings` array staggers when each guard activates (using `millis()`), so guards don't all appear at once — the level gets progressively harder as more patrols join. Reversing direction is handled simply by flipping the speed to negative.

### Lives System with Respawn

```java
void guardCollision() {
  for (int i = 0; i < 5; i++) {
    if (millis() > guardTimings[i]) {
      if (int(dist(mainCharacterX + 25, mainCharacterY + 35,
                   guardX[i] + 25, guardY[i] + 35)) < 65) {
        stealthLives--;

        if (stealthLives <= 0) {
          caught = true; // Game Over
        } else {
          // Teleport back to spawn with lives remaining
          mainCharacterX = 580;
          mainCharacterY = 180;
        }
      }
    }
  }
}
```

Rather than instant death on first contact, the player gets 3 lives. Each hit teleports them back to the start so they can try again — but guards don't reset, so re-navigating with fewer lives is genuinely harder.

## Part 7: The End Screen — Scrolling Credits

Once all four keys are collected, the final door unlocks. Walking through it triggers a scrolling text credits sequence based on the short story *"The Player"* by Naomi Alderman, with the player's name dynamically inserted throughout.

```java
void initEndScreen() {
  scrollY = height; // Start text off-screen below

  String pName = (line != null && line.length() > 0) ? line : "Player";
  fullPoem = fullPoem.replaceAll("PLAYERNAME", pName);
}

void playEndScreen() {
  background(0);
  fill(0, 170, 170);
  textSize(24);
  textAlign(CENTER, TOP);

  text(fullPoem, 100, scrollY, width - 200, 15000);
  scrollY -= 0.8; // Scroll up at 0.8 pixels per frame
}
```

The poem string uses `"PLAYERNAME"` as a placeholder throughout, and `replaceAll()` swaps every instance with the real name from the file. The scroll speed of `0.8` was chosen to match reading pace while still feeling cinematic.

## Key Takeaways

Building this project reinforced a few patterns that are worth keeping in mind for any Processing game:

**Use a single screen-state integer.** Routing all logic through one `screen` variable makes it trivial to add, remove, or reorganize game modes without restructuring the whole program.

**Separate axes in collision detection.** Testing X and Y independently prevents getting stuck on corners and gives the player a much smoother experience.

**Use `millis()` for staggered events.** Rather than adding complexity with multiple timers, `millis()` gives you a single always-running clock you can compare against stored trigger times.

**Distance-based hitboxes feel more forgiving.** For circular or vehicle-shaped objects, `dist()` between centers tends to feel fairer than tight rectangle overlaps.

**File I/O adds personality.** Even something as simple as reading a name from a file and printing it in the credits makes the game feel personal and complete.
